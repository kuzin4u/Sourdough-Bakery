# Монорепозиторий на Render: бот + сайт из подпапок одного репозитория

Если бот и сайт лежат в **одном** GitHub-репозитории (структура как в этом архиве:
`telegram-bot-python/`, `website/` и т.д.) — есть два способа сказать Render, из какой
подпапки собирать каждый сервис.

## Способ А — Blueprint, оба сервиса одним файлом (рекомендуется)

В этом архиве уже лежит готовый `render.yaml` **в корне** — он описывает сразу оба
сервиса и связывает их между собой автоматически: ключевая строка `rootDir` говорит
Render, из какой подпапки собирать каждый сервис, а блок `fromService` в сервисе сайта
сам подставляет адрес уже задеплоенного бота — вписывать его руками не нужно.

```yaml
services:
  - type: web
    name: sourdough-bakery-bot
    rootDir: telegram-bot-python   # ← вот это и есть привязка к подпапке
    ...
  - type: web
    name: sourdough-bakery-website
    rootDir: website                # ← и здесь тоже
    envVars:
      - key: BOT_HOST
        fromService:                # ← сайт сам узнаёт адрес бота
          type: web
          name: sourdough-bakery-bot
          property: host
```

### Как развернуть

1. Положите весь репозиторий (с этим `render.yaml` в корне) на GitHub — структура
   должна быть именно такой: `render.yaml`, `telegram-bot-python/`, `website/` —
   всё рядом, на одном уровне.
2. [render.com](https://render.com) → **New → Blueprint** → выберите репозиторий.
3. Render найдёт `render.yaml`, покажет оба сервиса (`sourdough-bakery-bot` и
   `sourdough-bakery-website`) и предложит создать их разом.
4. Заполните секреты с `sync: false` (токен бота, ключ Anthropic и т.д.) — Render
   сам спросит их в процессе создания.
5. **Apply** — Render развернёт оба сервиса. Сайт получит адрес бота автоматически,
   как только бот задеплоится (может понадобиться один Redeploy сайта, если сайт
   собрался раньше бота и `BOT_HOST` ещё не был доступен — Manual Deploy решает это).

### Если используете Node-версию бота вместо Python

Замените в `render.yaml` блок `sourdough-bakery-bot` на содержимое
`telegram-bot-node/render.yaml`, поменяв `rootDir` на `telegram-bot-node` и сверив
имя сервиса в блоке `fromService` сайта (`name: sourdough-bakery-bot` должно совпадать
с тем именем, что вы дали Node-сервису).

## Способ Б — вручную, без Blueprint, через поле Root Directory

Не хотите использовать единый `render.yaml` — можно создать оба сервиса вручную,
указав подпапку в настройках каждого:

1. **New → Web Service** → выберите репозиторий
2. В разделе **Root Directory** впишите `telegram-bot-python` (или `telegram-bot-node`)
3. Build/Start Command оставьте как в собственном `render.yaml` этой папки
   (`pip install -r requirements.txt` / `python main.py`)
4. Заполните Environment переменные вручную
5. Deploy → дождитесь адреса бота
6. **New → Static Site** → тот же репозиторий → **Root Directory** = `website`
7. Build Command: `bash build.sh`, Publish Directory: `.`
8. В Environment сайта впишите `BOT_API_URL` = адрес бота из шага 5 — **вручную**,
   этим способом автоматическая подстановка через `fromService` недоступна (она
   работает только внутри Blueprint, то есть только в Способе А)

Этот способ чуть дольше (два отдельных создания сервиса вместо одного Apply), но
проще для понимания, если Blueprint кажется избыточным.

---

## Какой способ выбрать

- **Способ А** — если планируете часто пересоздавать/клонировать окружение (например,
  тестовое и продакшен), или просто хотите минимум ручных действий.
- **Способ Б** — если уже привыкли создавать сервисы через интерфейс Render, или
  каждый сервис разворачиваете в разное время (не одновременно).

Оба способа приводят к одному и тому же результату — два независимых сервиса на
Render, сайт знает адрес бота. Технической разницы в работе самого приложения нет.
