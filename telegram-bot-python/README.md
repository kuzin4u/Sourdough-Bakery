# Telegram-бот пекарни на закваске (Python)

Полный функциональный аналог Node-версии: каталог, общая с сайтом корзина по телефону,
статусы заказов, стоп-лист, предзаказ на день выпечки, подписки, промокоды/лояльность,
онлайн-оплата (опционально), запрос отзыва, админ-команды, расширенная база знаний FAQ.

## 1. Команды бота

**Для покупателя:**
| Команда | Что делает |
|---|---|
| `/start` | приветствие |
| `/menu` | каталог (при первом обращении спросит телефон для синхронизации с сайтом) |
| `/cart` | показать текущую корзину |
| `/phone` | изменить номер телефона, привязанный к этому чату (для синхронизации с сайтом) |
| `/orders` | посмотреть свои последние заказы — из обоих каналов, по номеру телефона |
| `/subscribe` | оформить содержимое корзины как еженедельную подписку |
| `/unsubscribe` | отменить подписку |
| `/stock` | какие виды хлеба сегодня в наличии |

**Для администратора** (только `ADMIN_CHAT_ID`):
| Команда | Что делает |
|---|---|
| `/stop <id>` / `/instock <id>` | стоп-лист (id: `wheat`, `rye`, `wholegrain`, `seeds`) |
| `/addpromo КОД percent 10` | создать промокод |
| `/promos` | список промокодов |
| `/status <orderId> <статус>` | изменить статус заказа (или кнопки под уведомлением) |
| `/today_orders` | заказы за сегодня |
| `/all_orders [N]` | последние N заказов за всё время (по умолчанию 20) — не только за сегодня |
| `/stats` | сводная статистика |

Логика статусов, лояльности, подписок, предзаказа на день выпечки и онлайн-оплаты —
полностью идентична Node-версии. Подробное описание каждой механики — см. README
Node-бота (`telegram-bot/README.md`, раздел 2), здесь не повторяем, чтобы не дублировать.

## 2. База знаний FAQ

В системный промпт встроена сведённая база знаний о продукте (состав, хранение, польза,
для кого подходит) — собрана из двух исходных FAQ-документов и **скорректирована**:
часть формулировок (например, про "пробiotики в хлебе" или конкретный возраст введения
в прикорм) была смягчена или убрана как медицински неточная или неподтверждённая. Полный
разбор расхождений — в отдельном файле `FAQ_unified.md` (передан отдельно), здесь —
только итоговая версия, которая попадает в промпт консультанта (`FAQ_KNOWLEDGE` в начале
`main.py`). Стоп-лист учитывается динамически: бот не предлагает то, что в стоп-листе,
даже если вопрос касается этого вида хлеба.

### 2.1 Безопасное уведомление о заказах с сайта

`POST /api/notify-order` — браузерный виджет передаёт сюда данные заказа, а сообщение в
Telegram отправляет сам сервер бота (через `send_telegram_message`, минуя объект PTB —
см. техническую заметку в разделе 5). Токен бота используется только здесь, на сервере,
и никогда не передаётся в браузер. Раньше виджет хранил токен в `localStorage` и слал
сообщение в Telegram напрямую — это было небезопасно: токен виден в каждом сетевом запросе
сайта и при любом доступе к браузеру администратора. Теперь токен покидает `.env` только
в сторону `api.telegram.org`, не в сторону браузера.

### 2.2 FAQ-чат для сайта

`POST /api/ask-faq` — прямой вызов Anthropic API из браузера виджета работает только
внутри превью Claude.ai (там платформа сама даёт доступ к ключу) — на реальном сайте
у браузера нет `ANTHROPIC_API_KEY`, и запрос падает с "Что-то пошло не так". Виджет
сам определяет это: при ошибке прямого вызова автоматически переключается на этот
серверный маршрут, если в настройках указан адрес сервера бота. Без адреса бота на
реальном сайте AI-чат работать не будет — это ожидаемое ограничение, не баг.

### 2.3 Незавершённый заказ и просмотр своих заказов

Прогресс оформления (дата выпечки, район, имя, адрес, оплата, промокод) сохраняется
в `drafts.json` на каждом шаге — переживает перезапуск бота. При следующем `/menu`
бот предложит продолжить или начать заново, вместо того чтобы тихо всё потерять. На
финальном экране перед подтверждением теперь есть кнопка ✏️ "Изменить" — перезапускает
оформление с выбора даты выпечки, корзина не теряется. Команда `/orders` показывает
последние 10 заказов клиента из **обоих** каналов (сайт + Telegram) с статусом.

## 3. Установка и запуск

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python main.py
```

Модель для FAQ-консультанта переключается переменной `CLAUDE_MODEL` в `.env` (по
умолчанию `claude-haiku-4-5-20251001` — дешевле, для простых вопросов хватает с
запасом; можно поставить `claude-sonnet-4-6` для более развёрнутых ответов, дороже
примерно в 3 раза).

## 4. Деплой на Render

Идентично Node-версии: бот раздаёт HTTP API (корзина, стоп-лист, webhook ЮKassa),
поэтому деплоится **только как Web Service**.

1. Репозиторий на GitHub → **New → Blueprint** (подхватит `render.yaml`) или вручную
   **New → Web Service**, runtime Python, build `pip install -r requirements.txt`,
   start `python main.py`.
2. Плана: **Starter (~$7/мес)** — без "засыпания"; **Free** — с задержкой после простоя.
3. Переменные окружения — см. `.env.example`.
4. Deploy → проверьте `<адрес>/api/health` → `{"status":"ok"}`.
5. `/start`, затем `/menu` в Telegram.
6. Тот же адрес — в настройки (⚙) браузерного виджета, поле "Адрес сервера бота".

## 5. Технические отличия от Node-версии

- Flask (не aiohttp/FastAPI) в отдельном потоке — `python-telegram-bot` асинхронный,
  Flask синхронный, поэтому HTTP-сервер и Telegram-бот живут в разных потоках.
  HTTP-сервер поднят через `waitress` (production WSGI), а не встроенный дев-сервер
  Flask (`flask_app.run()`) — тот не предназначен для постоянной работы и выводит
  предупреждение в логи при каждом старте.
- Доступ к общим данным (`carts`, `orders` и т.д.) защищён `threading.Lock` — это нужно
  именно из-за многопоточности (в Node такой защиты не требуется, там однопоточный
  event loop).
- Уведомления из потока планировщика и из Flask-вебхука ЮKassa идут через прямой HTTP-вызов
  `api.telegram.org/bot<token>/sendMessage`, а не через объект бота `python-telegram-bot` —
  потому что у PTB v20+ свой asyncio event loop, и вызывать его методы из чужого потока
  напрямую ненадёжно (типичная ошибка "attached to a different loop"). Внутри обычных
  Telegram-хендлеров (когда сообщение пришло от пользователя) всё работает через
  стандартный `context.bot.send_message`, как и положено.
- **Связь с Telegram — webhook (по умолчанию) с откатом на polling.** Бот сам выбирает
  режим: если задан `USE_WEBHOOK=true` и есть публичный адрес (`RENDER_EXTERNAL_URL`,
  Render подставляет сам) — поднимается webhook на маршруте `/telegram-webhook` того же
  Flask-сервера, с проверкой секретного заголовка (`WEBHOOK_SECRET`) от подделки запросов.
  Без этих условий — автоматический откат на `run_polling()`, как раньше. Подробности
  и устранение проблем — раздел 5.1 ниже.

### 5.1 Известные проблемы продакшена — уже исправлены в этой версии

При первом деплое на Render может попасться один из трёх классических багов. Все три
уже пофикшены в текущем `main.py`/`render.yaml`/`runtime.txt`, но стоит знать, на что
смотреть в логах, если что-то пойдёт не так на вашей конкретной конфигурации:

- **`RuntimeError: There is no current event loop` на Python 3.14.** Начиная с этой
  версии Python, `asyncio.get_event_loop()` больше не создаёт event loop автоматически —
  а `Application.run_polling()` внутри PTB на это рассчитывает. Исправлено двумя
  независимыми способами: (1) `runtime.txt` и `PYTHON_VERSION` в `render.yaml` закрепляют
  Python 3.12.7 вместо новейшей 3.14; (2) в коде перед `run_polling()` event loop создаётся
  вручную как защита, даже если версия Python всё же окажется новее. **Важно:** при
  деплое через Render Blueprint (`render.yaml`) файл `runtime.txt` иногда игнорируется —
  основной механизм фиксации версии — именно переменная `PYTHON_VERSION` в `render.yaml`.
- **`WARNING: This is a development server` в логах.** Встроенный сервер Flask не
  предназначен для постоянной работы. Исправлено переходом на `waitress` (см. выше) —
  после этого предупреждение должно полностью исчезнуть из логов.
- **`telegram.error.Conflict: terminated by other getUpdates request`.** Возникает,
  когда два процесса одновременно опрашивают Telegram с одним токеном — чаще всего это
  нормальный кратковременный эффект rolling-деплоя Render (старый инстанс ещё не успел
  погибнуть, новый уже стартовал) и проходит сам за 30–60 секунд. `drop_pending_updates=True`
  в `run_polling()` снижает шум от накопленных обновлений после рестарта. Если ошибка
  не исчезает сама — проверьте в Render Dashboard, что под этим ботом нет второго
  забытого сервиса с тем же `TELEGRAM_BOT_TOKEN`, и что Scaling → Number of Instances = 1.
  Переход на webhook (что и сделано по умолчанию в этой версии) также снижает частоту
  такого конфликта, хотя не является прямым "лечением" причины с двумя инстансами.



## 6. Важные ограничения

Все данные в простых JSON-файлах (не переживают деплой на бесплатном плане Render —
файловая система не персистентна), webhook ЮKassa без проверки подписи, состояние
диалогов в памяти процесса (сбрасывается при рестарте).
